# Future Strategy Architecture (FSA): Strategy Segregation Plan

## Overview
Transition the trading engine from a monolithic `config.py` structure to a modular, file-based "Strategy Plugin" architecture. Each trading strategy will exist as an independent JSON configuration file.

## Mandatory Database Traceability
Every data object generated by the pipeline (Recommendations, Positions, Scans, Backtests) **MUST** include the following metadata fields for auditability:

1. `strategy_id`: A unique identifier for the strategy (e.g., `EP_PATEL_01`).
2. `strategy_name`: The human-readable name (e.g., `Delayed Episodic Pivot`).
3. `strategy_filename`: The exact name of the source JSON file (e.g., `episodic_pivot.json`).
4. `strategy_path`: The relative path to the configuration file (e.g., `configs/strategies/episodic_pivot.json`).

## Folder Structure
```text
backend/
├── config.py (Core credentials & system settings only)
└── configs/
    └── strategies/
        ├── episodic_pivot.json
        ├── standard_swing.json
        └── mean_reversion.json
```

## Strategy JSON Schema Requirements
Each JSON file must encapsulate the following logic:
- **Scanner Configuration**: Chartink clauses or internal scanning logic.
- **Trading Gates**: Technical thresholds (ADX, SMA, Volatility).
- **Execution Params**: Stop-loss, Take-profit, and Pullback targets.
- **Scoring Weights**: Distribution between Technical, Fundamental, and Sector scores.

## Future Implementation Steps
1. Create `configs/strategies/` directory.
2. Build a `StrategyManager` class to dynamically load and validate JSON files using `os.listdir`.
3. Update `analyzer.py` and `main_orchestrator.py` to iterate over loaded strategies.
4. Update MongoDB handlers to ensure all records carry the mandatory strategy metadata.
